## Context

当前 mod 端的 `play_card` 执行链路优先复用 UI 拖拽式出牌流程。该路径会创建 `NMouseCardPlay` 一类界面对象，但在当前游戏版本里，只有真正完成拖拽、目标选择与落点判定后才会进入实际出牌逻辑。bridge 仅看到入口调用成功，就把结果回报为 accepted，导致 `/apply` 与 live snapshot 之间出现“回执成功、状态不变”的严重错配。

同时，战斗窗口识别与额外选牌判断里，部分代码会直接读取会触发格式化的 `Description`。当本地化变量或 selector 缺失时，运行时会抛出格式化错误并持续刷日志。问题虽然不一定直接阻断桥接线程，但会污染排障信号，并增加 live 调试成本。

## Goals / Non-Goals

**Goals:**
- 让 `play_card` 优先走可直接触发游戏逻辑的运行时入口，避免“假成功”。
- 让 `accepted` 与真实 live 状态推进绑定，至少能区分“已接收但未实际生效”。
- 让战斗窗口识别与选牌识别改用安全文本读取路径，不再因 description 诊断触发本地化异常刷屏。
- 保持现有 `/apply`、`/snapshot`、`/actions` 协议兼容，仅提升语义可靠性与日志质量。

**Non-Goals:**
- 不在这次变更中扩展新的动作类型。
- 不重做整套文本渲染架构，只修正会影响 live 对局和诊断的危险读取路径。
- 不尝试在 spec 中承诺所有卡牌 description 都完全无模板残留；本次重点是避免异常和错误成功语义。

## Decisions

### 1. `play_card` 优先使用直接运行时出牌入口

优先从卡牌实例上查找类似 `TryManualPlay(target)` 的直接逻辑入口，让 bridge 直接触发真实出牌结算，而不是先模拟 UI 拖拽。这样更符合 bridge 的“外部受控执行”语义，也更能跨过当前版本 UI 手势链路的异步依赖。

备选方案是继续修补 `NMouseCardPlay` / `NCardPlay` 的 UI 路径，但这条链路对拖拽状态、落点区域、目标确认依赖更强，版本波动也更大，不适合作为 bridge 的首选执行路径。因此只保留为兼容 fallback，而不是成功语义的依据。

### 2. `accepted` 必须与可观察状态推进绑定

动作请求进入执行队列并不等于动作已经真实生效。bridge 在 `play_card` 执行后需要检查至少一个稳定推进信号，例如 `decision_id`、`state_version`、手牌实例集合、能量、敌我生命/格挡，或进入新的额外选牌窗口。若没有任何推进，应降级为 `failed` 或等效可诊断结果，并附带执行阶段信息。

备选方案是保持当前“入口调用成功即 accepted”，由客户端自行轮询判断，但这会把桥接层最关键的执行真伪判断推给外部 runner，已经被 live 实测证明会直接拖死自动对局，因此不采用。

### 3. 战斗选牌识别改用安全描述来源

用于窗口识别的卡牌文本不需要最终渲染后的玩家可见说明，只需要足够稳定的“这是不是一张真实卡牌/选牌项”信号。因此这里应优先读取原始模板、已缓存文本或其他不会触发 `LocString` 格式化的字段；若拿不到，就直接保守降级，而不是强行访问 `Description`。

备选方案是继续读取最终 `Description` 并在异常处加大 try/catch，但这仍会频繁进入高成本异常路径，且日志噪音大，不如从根源上换成安全读取。

### 4. 说明格式化异常保留日志诊断，但不干扰主流程

对于仍可能出现的 description 解析失败，bridge 应记录带对象标识、字段来源与阶段的 warning，并按既有 fail-safe 逻辑继续构建 snapshot 或 actions。默认日志聚焦失败和降级路径，不再因为普通窗口识别或诊断探测而持续刷屏。

## Risks / Trade-offs

- [直接运行时入口在不同版本改名或签名变化] → 保留反射探测与 fallback，并在 action metadata / 日志中记录实际命中的 handler。
- [状态推进判定过严，误把合法但延迟结算的动作判成失败] → 采用短窗口轮询或多信号组合判定，允许进入新 `decision_id`、新选择窗口或资源变化任一命中即视为成功。
- [安全描述读取过于保守，降低部分选牌识别命中率] → 优先使用多种低风险字段组合，而不是单一文本来源；命中失败时保守返回过渡/未知窗口，不伪造可执行动作。
- [日志收敛后排障信息不足] → 失败路径仍保留对象标识、handler、阶段与异常摘要，必要时再通过显式 debug 开关扩展细节。
