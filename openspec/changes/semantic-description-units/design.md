## Context

当前 `Sts2RuntimeReflectionReader` 已经能从 `LocString`、运行时渲染文本、hover tip 与对象成员中恢复大部分说明内容，但渲染链路仍把很多动态变量当作“无语义数字”处理。结果是同一个数值在不同场景下可能丢失单位，例如商店移除服务的涨价幅度只剩 `25`，卡牌收益只剩 `2`，虽然游戏对象本身已经提供了变量来源、动态方法或 hover tip 线索。

这个问题横跨 shop service、shop card、relic、potion 等多条说明链路，不适合继续用单个对象的字符串补丁修复。需要把“变量值”提升为“带语义的变量值”，再在受控场景下做统一的描述归一化。

## Goals / Non-Goals

**Goals:**
- 在 description 提取链路中增加可复用的变量语义层，例如 `gold`、`energy`、`cards`、`hp`。
- 在商店服务、卡牌、遗物、药水等说明对象上复用统一的语义化后处理，补全缺失单位并保留游戏原始渲染优先级。
- 尽量依赖游戏运行时对象、动态变量、方法返回值与 hover tip，而不是针对某条中文文案做静态替换。
- 保留日志诊断，便于识别哪些说明仍然缺少可靠语义来源。

**Non-Goals:**
- 不重建完整本地化引擎，也不尝试覆盖所有 SmartFormat 语法扩展。
- 不为了单个对象引入独立 schema 字段；本次优先提升现有 `description` 的可读性。
- 不修改 `/apply`、动作协议或 agent 决策格式。

## Decisions

### 决策一：为描述变量增加 `semantic kind`，而不是继续只传裸整数
为 `DescriptionVariable` 的解析流程增加语义分类，来源包括占位符名、成员别名、调用方法名以及所在对象上下文。这样 `Amount` 可以在商店服务上下文中落到 `gold`，`Energy` 可以稳定落到 `energy`。

备选方案：
- 仅按渲染后的中文字符串补单位：实现快，但会绑定当前语言与具体文案，泛化差。
- 只依赖 glossary：glossary 能解释术语，但不能可靠说明一个裸数字应该补什么单位。

### 决策二：保留“游戏渲染优先”，在后处理阶段做最小语义补全
当游戏已经给出完整描述时，直接使用游戏文本；只有在渲染结果中仍出现裸数字、图标路径或缺失单位时，才进入 bridge 归一化。这样可以减少偏离游戏原文的风险。

备选方案：
- 一律使用模板加本地渲染：控制力更强，但更容易与游戏真实展示不一致。
- 一律使用运行时字符串：会保留 `res://...energy_icon.png` 或 `增加25。` 这类对 agent 不友好的结果。

### 决策三：按场景启用语义补全，避免全局误伤
商店服务、商店商品描述、战斗外预览说明等场景对可读性要求高，且缺少 UI 图标上下文，适合启用单位补全。对于战斗中已经完整渲染的文本，只在明确识别到资源路径或裸数字变量时处理，避免把普通数值误写成错误单位。

备选方案：
- 全局对所有数字补单位：风险过高，容易误伤层数、倍率、命中次数等字段。
- 每个对象单独写规则：维护成本高，后续扩展到 event、shop、relic、potion 会重复。

### 决策四：无法可靠确定语义时只记日志，不做猜测性增强
如果变量来源模糊、上下文不足或多个单位候选冲突，bridge 继续返回当前可得文本，并把诊断写入日志，供后续针对具体对象补充来源解析。这样能保证输出稳定性优先于“看起来更聪明”的猜测。

## Risks / Trade-offs

- [语义误判] -> 通过对象上下文与变量来源双重约束，只在高置信度场景补单位；低置信度时保留原文并记日志。
- [输出与游戏原文不完全一致] -> 明确以游戏渲染优先，只对游戏未补全的语义缺口做最小增强。
- [规则逐渐膨胀] -> 把补全逻辑收敛到统一的变量语义层，而不是为每个卡牌或服务单独写字符串替换。
- [后续 mod 或语言包兼容性] -> 尽量基于运行时对象、变量名与 canonical 术语，不依赖中文文案常量。
